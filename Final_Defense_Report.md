CHAPTER 1: INTRODUCTION

1.1 Background

The Government of Nepal continues to rely extensively on paper-based documentation across key sectors such as land registration, judiciary records, education, civil administration, and local municipalities. A large portion of these records are written in Nepali using the Devanagari script, either in handwritten or printed form. The digitization of such documents into searchable, editable, and machine-readable text is a long-standing challenge for effective e-governance, digital archiving, and data-driven decision-making.

Optical Character Recognition (OCR) is the technology that enables the automatic conversion of scanned document images into digital text. While OCR systems for printed English and other high-resource languages have achieved near-human accuracy, OCR for Nepali Devanagari documents — particularly handwritten ones — remains comparatively unreliable. This limitation arises from the structural complexity of the Devanagari script, which combines a connecting head stroke (shirorekha), dependent vowel signs (matras), halant-joined conjunct consonants, and several visually similar base characters, together with wide variation in individual handwriting style and scan quality [1], [2]. A script-independent handwritten-recognition study published in 2021 makes the same point formally: recognition networks that are not designed with the specific glyph geometry of a script in mind transfer poorly to scripts such as Devanagari that combine strokes above and below a shared headline [2].

Traditional OCR approaches for Devanagari-family scripts have largely focused on isolated character recognition using handcrafted features or single-glyph Convolutional Neural Networks (CNNs). CNN-based classifiers report very high accuracy when the recognition task is reduced to one isolated, centred glyph — a 2024 CNN architecture trained on the standard 92,000-image Devanagari Handwritten Character Dataset (DHCD), for instance, reports a test accuracy above 99% [3] — but this framing does not carry over to full documents. A single-character classifier has no mechanism for reading matras, conjuncts, punctuation, or the way the shirorekha visually joins the letters of a word into one connected stroke; it can only ever emit one base glyph per input crop, which is a structurally different (and easier) problem than reading a line of continuous handwriting [2], [4].

Recent advances in deep learning, particularly sequence-based Convolutional Recurrent Neural Networks trained with Connectionist Temporal Classification (CTC) loss, and Transformer-based architectures, have significantly improved OCR performance for complex and low-resource scripts. The Transformer-based TrOCR model, for example, combines a pre-trained Vision Transformer image encoder with a pre-trained text Transformer decoder and reports state-of-the-art results on printed, handwritten, and scene-text benchmarks without hand-designed pre- or post-processing [5]. Because TrOCR's decoder generates text token-by-token with self-attention over the full decoded prefix, it is naturally able to model the sequential and contextual dependencies between characters within a word — the very dependencies that a single-glyph CNN classifier cannot represent — which motivated its adoption as the sequence-modelling counterpart to the CNN baseline in this project. Recent Devanagari-specific work reinforces the same conclusion from the other direction: a 2025 study on post-OCR error correction for low-resource Devanagari languages shows that even a reasonably accurate character-level recognizer produces text with residual errors that are systematically correctable once contextual, sequence-level information is brought to bear [4], and a 2025 survey of OCR for Devanagari-script languages identifies segmentation of joined and modified characters as the dominant open problem across the family of Devanagari OCR systems, Nepali included [6].

This project builds an end-to-end Devanagari OCR system in two stages that mirror this literature. The first stage trains and evaluates a CNN baseline (this project's CRNN track) and a Transformer baseline (TrOCR) on the isolated-character task, to establish a fair, apples-to-apples comparison on a controlled, balanced dataset before either model is asked to handle a harder problem. The second stage extends the Transformer branch from single characters to whole handwritten lines and printed document lines — the actual target described in the project proposal — using a synthetic-data pretraining strategy adapted to Nepali, together with a line-segmentation front end that crops whole text lines instead of individual characters so that the shirorekha-joined structure of a word is never broken apart before recognition.

1.2 Problem Statement

Converting handwritten and scanned Nepali documents into usable digital text remains difficult in practice. Most publicly available OCR systems are optimized for printed Latin-script text and are unreliable on Devanagari, particularly on handwritten Devanagari, because the script's modifiers, conjuncts, and shirorekha connections are not represented in the training data or model design of general-purpose OCR engines. As a result, a large volume of Nepali government and institutional documents remains inaccessible in searchable digital form, and organizations fall back on manual data entry, which is slow, error-prone, and does not scale.

Testing an isolated single-character CNN/Transformer classifier on a real, continuously handwritten Nepali page in this project made the underlying problem concrete rather than theoretical: character-segmentation can only reliably cut at whitespace, but the shirorekha physically joins the letters of a word into one connected stroke, so an entire word becomes a single blob that is then handed to a model that can only output one base glyph. A photographed line reading "मेरो नाम संस्कृति हो।" was, under this approach, recognized as the nonsense string "त्रगल३४ ज" — confirming that the limitation is structural, not a tunable defect of a particular character classifier, and that full-document recognition requires a sequence model trained on whole word or line images rather than a better single-character classifier [2].

There is accordingly a need for (1) a reliable comparative baseline that measures how a CNN-CTC architecture and a Transformer architecture each perform on the same, controlled Devanagari character-recognition task, and (2) a document-level recognition path — built on the stronger of the two architectures — that reads real lines of Devanagari text, including matras, conjuncts, and mixed Devanagari/Latin content of the kind found on real printed forms, rather than isolated glyphs.

1.3 Objectives

The objectives of this project are:

i. To build a CNN-based OCR model and a Transformer-based OCR model for Devanagari characters, and compare which one recognizes them more accurately.

ii. To extend the better-performing model from single characters to full lines of handwritten and printed Nepali text, and use it in a working web application that turns a scanned document into editable Unicode text.

1.4 Significance of the research

The Government of Nepal and its constituent institutions depend heavily on Nepali-language documents for administrative, legal, historical, and archival purposes, and a large proportion of these documents exist only in handwritten or scanned paper form. This dependence on physical records limits the effectiveness of e-governance initiatives and leaves institutional memory exposed to permanent loss from fire, flood, or physical damage to the offices where the records are held [6].

This research is significant in three respects. First, it produces a controlled, reproducible comparison between a CNN-CTC architecture and a Transformer architecture on the same balanced Devanagari dataset, contributing an evidence-based answer — rather than an assumed one — to which family of architectures is better suited to this script, consistent with the broader literature's finding that Transformer decoders' self-attention over the full output prefix is well matched to scripts with strong intra-word dependencies such as Devanagari [5]. Second, it moves beyond isolated-character recognition, which is the level at which almost all prior Nepali/Devanagari OCR work — including the DHCD-benchmarked CNN literature [3] — has been evaluated, toward document-level recognition of real handwritten and printed lines, which is what practical digitization actually requires. Third, it demonstrates the result in a working, end-to-end web application rather than only as an offline evaluation script, so the system's practical usefulness — and its remaining limitations — can be assessed directly rather than inferred from benchmark numbers alone.

1.5 Scope and Limitation

The scope of this project covers Devanagari character-level OCR (CNN and Transformer baselines, trained and evaluated on the standard 92,000-image, 46-class Devanagari Handwritten Character Dataset) and an extension to document/line-level OCR using a Transformer model trained primarily on synthetically generated word- and line-images, since large labelled corpora of handwritten Nepali lines do not exist publicly. The system accepts scanned images and multi-page PDF documents as input and produces editable Unicode Devanagari text as output, exportable as plain text, DOCX, and a searchable PDF with an invisible, selectable text layer.

Several limitations follow directly from this scope. Recognition performance at the document level is trained on synthetic data rather than large-scale real handwritten line corpora, so accuracy on real scanned pages — particularly dense, tabular, or heavily degraded forms — is expected to be lower than on the clean, balanced character-level benchmark; this gap is a genuine domain limitation, not merely an implementation defect, and is examined directly in Chapter 4. The document-level Transformer model is also more computationally demanding to train and to serve than the CNN baseline, which constrains how quickly it can be iterated on within the available GPU quota (a practical constraint encountered directly during this project, discussed in Chapter 3). Finally, the system's Unicode output for genuinely ambiguous or very low-quality handwriting may still require manual verification or correction before it is treated as an authoritative digital record.

CHAPTER 2: LITERATURE REVIEW

2.1 Overview of Existing Systems

Handwritten and printed character recognition for Devanagari-family scripts has been studied for over a decade, but the techniques used have shifted through three broad generations that roughly parallel the wider evolution of computer-vision OCR: handcrafted-feature classifiers, CNN-based deep classifiers, and — most recently — sequence and attention models capable of reading beyond a single glyph. This section traces that evolution and situates the CNN and Transformer models built in this project within it.

2.1.1 OCR Evolution and CNN-Based OCR

Early Devanagari recognition systems relied on handcrafted, geometry-based descriptors rather than learned features. Pal and Chanda combined gradient and curvature features with a hybrid Support Vector Machine (SVM) and Modified Quadratic Discriminant Function (MQDF) classifier and achieved high accuracy on isolated offline handwritten Devanagari characters, but reported persistent confusion between visually similar character pairs — a limitation of describing a glyph by a fixed, hand-designed feature vector rather than letting a model learn which visual cues actually distinguish confusable characters [1]. The shift to Convolutional Neural Networks (CNNs) addressed this directly: a CNN learns its own hierarchy of visual features directly from pixel data, starting with low-level edges and textures in its early layers and composing them into progressively more complex, class-specific stroke patterns in deeper layers. This end-to-end feature learning is what makes CNNs effective for handwritten OCR, where the handcrafted features used in earlier work often failed to separate characters that differ only in a small curve or an added stroke. Recent CNN work on the standard 92,000-image Devanagari Handwritten Character Dataset (DHCD) illustrates how far this generation of models has been pushed: HCR-Net, a script-independent transfer-learning CNN, and MallaNet, a residual branch-merge CNN with homogeneous filter capsules purpose-built for Devanagari, both report test accuracies close to or above 99% on the isolated 46-class character task [2], [3]. This project's own CRNN baseline (Chapter 3) sits in the same lineage: it uses a CNN feature extractor for exactly this reason, and its 98.67% test accuracy on the same DHCD-style task is consistent with what this generation of architecture is known to achieve.

The consistent 99%-class accuracy of modern CNNs on isolated Devanagari glyphs is, at the same time, the key limitation this project had to work around. All of the systems above are evaluated on a single, pre-cropped, pre-centred character; none of them are asked to locate, segment, and order the characters of a real handwritten word or line. A CNN classifier has no internal mechanism for handling a matra attached to a base consonant, a halant-joined conjunct spanning two or three base glyphs, or the shirorekha that visually joins the letters of an entire word into one connected stroke — it can only accept one cropped glyph and emit one label. This is a well-known ceiling in the literature, not a weakness specific to any one paper: HCR-Net's authors frame their contribution explicitly as script-independent character recognition, not document recognition [2], and it is exactly this ceiling that motivates the CRNN-CTC and Transformer sequence architectures discussed next.

2.1.2 CRNN and Sequence Modelling with CTC

The Convolutional Recurrent Neural Network (CRNN), introduced by Shi, Bai, and Yao, was the first architecture to demonstrate that a single, end-to-end trainable network could read an entire line of text without first segmenting it into individual characters [7]. A CRNN passes a text-line image through a CNN to obtain a sequence of feature columns, feeds that sequence through Bidirectional Long Short-Term Memory (BiLSTM) layers to model how each character's visual features are influenced by its neighbours, and aligns the resulting per-column predictions to the (much shorter) ground-truth text using Connectionist Temporal Classification (CTC) loss. CTC's contribution is specifically to the segmentation problem: rather than requiring a labelled boundary for every character — which is expensive to annotate and does not exist for scripts like Devanagari where letters are visually joined — CTC allows the network to predict a label (or a "blank") at every time step and then defines a loss that sums over every possible alignment between that raw prediction sequence and the target text, collapsing repeated predictions and removing blanks. This is why a CRNN-CTC model is described as segmentation-free: it never needs to be told where one character ends and the next begins.

BiLSTM layers are used specifically (rather than a plain feed-forward layer) because Devanagari text carries dependencies that extend beyond a single glyph's own pixels: a matra's correct reading depends on the consonant it is attached to, and a conjunct's correct reading depends on both halves of the joined stroke. An LSTM cell processes a sequence and retains information across time steps through a gated memory cell, letting the network carry forward context from earlier in a word to influence how a later part of the same word is decoded, and a bidirectional LSTM does this in both reading directions simultaneously so a character's prediction can draw on context that comes after it as well as before it. Long Short-Term Memory itself was designed to solve the vanishing-gradient problem that plain recurrent networks suffer from over long sequences, using input, forget, and output gates to control what information a memory cell keeps, discards, or exposes at each step — the forget gate specifically was added to let the network learn to reset a memory cell's state when a sequence boundary is reached, rather than accumulate irrelevant context indefinitely [8].

2.1.3 Transformer-Based OCR and Vision Transformer

The Transformer architecture replaced recurrence with self-attention: instead of processing a sequence one step at a time and carrying state forward through a memory cell, a Transformer layer lets every position in a sequence directly attend to every other position in a single operation, weighting how much each other position should influence it. This removes the sequential bottleneck of recurrent networks — attention over the whole sequence is computed in parallel — and, more importantly for OCR, gives the model a direct, unmediated path between any two positions in a word or line, rather than a path that must pass information through every intermediate time step as an LSTM does. For a script like Devanagari, where a matra can visually sit to the left of the consonant it modifies (breaking simple left-to-right locality) and a conjunct can visually compress two or three letters into one shape, this direct long-range connectivity is a better structural fit than a purely sequential recurrence.

The Vision Transformer (ViT), introduced by Dosovitskiy et al., extended this idea to images by treating an image as a sequence rather than a grid: the image is cut into fixed-size, non-overlapping patches (typically 16×16 pixels), each patch is flattened and linearly projected into an embedding vector, a learned positional encoding is added so the model knows each patch's original location, and the resulting sequence of patch embeddings is processed by a standard Transformer encoder exactly as if it were a sequence of word tokens [9]. Positional encoding is necessary specifically because self-attention itself has no notion of order — without it, a Transformer would treat an image (or a sentence) as an unordered set of tokens.

TrOCR, the Transformer-based OCR model used in this project's document-recognition track, combines a ViT-style image encoder with a Transformer text decoder, both initialised from large-scale pre-training rather than trained from scratch, and reports state-of-the-art results on printed, handwritten, and scene-text benchmarks without any hand-designed pre- or post-processing [10]. Decoding in TrOCR is autoregressive: the decoder generates one output token at a time, and at each step its self-attention looks back over every token it has already generated while its cross-attention looks across the full sequence of encoded image patches, so each new character prediction is conditioned on everything decoded so far and on the whole image, not just a local crop. This is the same mechanism this project relies on when extending TrOCR from single characters to full lines in Chapter 3: because the decoder already attends over its own generated prefix, it can be trained on a longer target string (a whole word or line) without any architectural change, only a longer maximum generation length.

A more recent Transformer OCR variant, PARSeq, illustrates a further refinement in how the decoder is trained: rather than always decoding strictly left-to-right, PARSeq trains an ensemble of autoregressive orderings using permutation language modelling, which lets the same model be used either as a fast, parallel non-autoregressive reader or as a more accurate iterative, context-refining autoregressive reader, and it reports state-of-the-art scene-text accuracy under both regimes [11]. This shows that the specific decoding order used by a Transformer decoder — not just its architecture — is itself an active area of OCR research, and is a natural direction for future refinement of the line-level Nepali model built in this project (Chapter 5).

2.1.4 Nepali-Specific OCR Research

Because this project targets Nepali handwriting specifically rather than Devanagari in general, Nepali-language OCR research is the most directly relevant body of literature, and much of it originates from Bal Krishna Bal's Information and Language Processing Research Lab (ILPRL) at Kathmandu University, which has worked on Nepali OCR, machine translation, and text-to-speech since the mid-2000s. Hengaju and Bal addressed the specific, practical problem of adapting a general-purpose engine to Nepali: rather than retraining Tesseract-OCR, they designed an eight-step image-preprocessing pipeline (deskewing, denoising, and contrast/threshold correction tuned to Nepali text images) and showed that this preprocessing alone raised Tesseract's recognition accuracy from 90.69%, 54.34%, and 38.45% to 94.84%, 71.15%, and 51.21% on high-, medium-, and low-quality Nepali document images respectively [14]. This result is a direct precedent for the preprocessing pipeline built in this project's own methodology (Chapter 3): it demonstrates that, for Nepali text specifically, a large share of the achievable accuracy gain comes from image conditioning before the recognition model ever sees the input, independent of which recognition architecture is used downstream.

More recently, Bal's lab moved from adapting an existing OCR engine to building a Nepali-specific sequence model from scratch. Nakarmi, Sthapit, Shakya, Chulyadyo, and Bal trained a CRNN-CTC architecture directly on Nepal script text lines, following the same CNN-BiLSTM-CTC design used for scene text elsewhere in the literature [7] but tuned to Devanagari-based Nepali text rather than Latin text, and reported it as a segmentation-free line-recognition baseline specifically for the Nepali OCR community [15]. This is, in effect, the same architectural family as this project's CRNN baseline, applied to the same target language, and its existence confirms that CRNN-CTC is an established, credible baseline choice for Nepali OCR rather than an arbitrary one. On the character-recognition side, Acharya, Dhungel, and Jha built a Nepali Handwritten Character Recognition System using a CNN classifier, reporting the same broad pattern seen in the wider Devanagari CNN literature: strong accuracy on isolated, pre-segmented Nepali characters, without addressing full-word or full-line recognition [16].

The clearest precedent for the Transformer half of this project's comparison is Hasan, Dhakal, Mehedi, and Rasel's 2024 study, which built and evaluated a Transformer-based encoder-decoder OCR model on Nepali and Bengali optical text images, covering both handwritten and printed samples [17]. Its result — that a Transformer encoder-decoder can be trained end-to-end on Devanagari-family script images without hand-designed segmentation, mirroring TrOCR's approach [10] — directly supports this project's decision to extend TrOCR from single Devanagari characters to full Nepali document lines rather than treating the Transformer as untested for this script family. Finally, Shahi and Sitaula's broader review of natural language processing for Nepali text situates OCR within the wider resource landscape for the language: it documents that the scarcity of large, standardized, publicly available Nepali-language datasets — for OCR as much as for other NLP tasks — is a structural bottleneck across the field, not one specific to any single project or model [18]. This is the same data scarcity that motivated this project's use of synthetically generated Nepali word- and line-images to pretrain its document-level TrOCR model (Chapter 3), rather than relying on a real, human-annotated corpus that does not yet exist at the required scale.

2.1.5 Commercial and General-Purpose OCR Engines

Alongside architecture research, several general-purpose OCR engines are widely used in practice and are relevant as a baseline for how well "off-the-shelf" OCR already handles Devanagari. Tesseract is an open-source OCR engine originally built around a CRNN-style pipeline and trained predominantly on printed Latin-script text; a 2024–2025 zero-shot evaluation on low-resourced non-Latin scripts found it "profoundly ineffective" outside its well-supported languages, with a mean Character Error Rate around 0.86 on unfamiliar scripts, reflecting how little of its training distribution resembles Devanagari handwriting [12]. EasyOCR, a deep-learning OCR toolkit covering roughly 80 languages including Devanagari, performed noticeably better in the same evaluation (mean CER around 0.47) but still fell well short of being reliable for unconstrained use [12]. Google's Cloud Vision OCR API substantially outperformed both open-source engines on Devanagari in the same comparison, which the literature attributes to the scale and diversity of its training data — including a large corpus of real-world Indic-script images — rather than to any single architectural advantage [12]. Microsoft's Azure AI Vision OCR occupies a similar commercial tier to Google's offering, with broad multilingual printed-text support but comparatively little published, independent benchmarking specifically on handwritten Devanagari. The consistent pattern across all four engines is that none of them is designed or tuned specifically for handwritten Nepali government-document text, which is precisely the gap this project's document-level TrOCR model (Chapter 3) targets by training on Nepali-specific synthetic data rather than relying on a general-purpose multilingual engine.

2.3 Research Gaps

2.3.1 Lack of Generalization: Systems trained and benchmarked on one controlled dataset — most commonly the isolated-character DHCD benchmark — do not automatically generalize to real, continuously written documents, scanned forms, or mixed Devanagari/Latin content. The 99%-class accuracy reported for CNN classifiers on DHCD [2], [3] and the much lower, inconsistent accuracy that general-purpose engines achieve on real Devanagari text in the wild [12] are two measurements of the same underlying gap.

2.3.2 Segmentation Errors: Every recognition pipeline that depends on cutting a page into individual character images before classification inherits the same weakness: the shirorekha and halant-joined conjuncts in Devanagari do not leave clean character boundaries to cut along, so over-segmentation and under-segmentation are unavoidable rather than incidental bugs. Pant and Bal identify this explicitly as the dominant remaining error source in their hybrid Nepali OCR system even after a roughly 10% accuracy improvement from combining word- and character-level recognition [13].

2.3.3 Limited Handling of Complex Characters: Most published Devanagari CNN classifiers, including the highest-accuracy ones reviewed above, are trained and evaluated on the base set of consonants and digits; conjuncts, ligatures, and rare compound forms are typically under-represented in benchmark datasets and correspondingly under-served by the resulting models.

2.3.4 Dependence on Large Labelled Datasets: Transformer-based OCR models are markedly more data-hungry than CNN classifiers, since they are not pre-biased toward local, translation-invariant features the way a CNN is [9]. Large, labelled corpora of real handwritten Nepali lines or documents — as opposed to isolated characters — do not currently exist in the public domain; Shahi and Sitaula's review of Nepali NLP identifies this dataset scarcity as a structural, language-wide bottleneck rather than a gap specific to OCR [18], which is the direct motivation for the synthetic-data pretraining strategy adopted in this project's document-level track (Chapter 3).

2.3.5 High Computational Requirements: Transformer OCR models are substantially larger and more expensive to train and serve than CNN classifiers of comparable task scope, which limits how quickly they can be iterated on without dedicated GPU infrastructure — a constraint this project encountered directly when a cloud GPU session's usage quota was exhausted mid-training (Chapter 3).

2.4 Summary

The literature shows a consistent progression from handcrafted features, to CNN classifiers that learn their own visual features but are confined to isolated glyphs, to sequence models (CRNN-CTC) and Transformer models (TrOCR, PARSeq) that can read variable-length text without prior character segmentation. Nepali-specific work spanning nearly two decades of research at Kathmandu University's ILPRL — from preprocessing-based improvements to an existing engine [14], to a dedicated Nepali CRNN-CTC baseline [15], to isolated-character CNN classifiers [16] — mirrors this same progression, and a 2024 Transformer OCR study on Nepali and Bengali text [17] confirms that the Transformer branch of this progression is applicable to Devanagari-family scripts specifically, not only to Latin scene text. Taken together, the literature confirms that the remaining open problem for Nepali OCR is not raw single-character accuracy — which is already close to saturated at roughly 99% on standard benchmarks [2], [3] — but generalization from isolated characters to real, connected, document-level text, compounded by a persistent shortage of large, public, real handwritten Nepali datasets [18]. This project's methodology (Chapter 3) is built directly around closing that specific gap: a controlled CNN-vs-Transformer comparison at the character level, followed by an extension of the stronger architecture to document-level Nepali text using synthetic-data pretraining and line-level segmentation.

CHAPTER 3: METHODOLOGY

This chapter describes how the two OCR pipelines introduced in Chapter 2 — a CNN-based CRNN model and a Transformer-based TrOCR model — were built, trained, evaluated, compared, and finally deployed as a working document-digitization system. The chapter follows the order in which the work was actually carried out: dataset preparation, preprocessing, the two model architectures, training (including two bugs that were found and fixed during TrOCR training), the extension from isolated characters to full document lines, the web application and API that expose the trained models, evaluation methodology, the experimental environment, the technology stack, system design, and testing.

3.1 Dataset

The character-level comparison used the Devanagari Handwritten Character Dataset (DHCD): 92,000 grayscale images across 46 classes (36 consonants and 10 digits), each image 64×64 pixels. The dataset is perfectly balanced at 2,000 images per class, which was confirmed by exploratory data analysis (class counts, image-size distribution, and sample grids) before any model was trained, so that the very high accuracy reported in Chapter 4 could not be an artefact of class imbalance. A reproducible 80/10/10 train/validation/test split (73,609 / 9,191 / 9,200 images) was generated with a fixed random seed and persisted to a split-index file, so that the CNN and Transformer models were trained and evaluated on exactly the same partitions — a requirement for the comparison in Section 3.9 to be meaningful. The 46-class label set and the CTC blank symbol were fixed in a single character-set definition file shared by every training and inference script, so that class indices could not silently drift between the CRNN and TrOCR pipelines.

For the document-level extension (Section 3.6), no equivalent real, labelled corpus of handwritten Nepali lines or full pages exists publicly, consistent with the dataset-scarcity gap identified in Chapter 2 [18]. A synthetic word- and line-image dataset was therefore generated specifically for this project, described in Section 3.6.1.

3.2 Data Preprocessing

A single preprocessing pipeline was implemented once and imported by every training and inference script — the CRNN trainer, the TrOCR trainer, the backend API, and the web application all call the same function rather than each re-implementing the steps — specifically to eliminate train/serve skew, the class of bug where a model performs well in evaluation but poorly in production because the live input is processed slightly differently from the training input. The pipeline performs, in order: (1) conversion to grayscale; (2) Gaussian blur to suppress scan and camera noise; (3) adaptive thresholding (Gaussian-weighted, inverse binary) to binarize the image under uneven local lighting rather than a single global threshold, which handles the uneven illumination typical of phone-camera document photos; (4) skew correction, using the minimum-area bounding rectangle of the foreground pixels to estimate and undo any rotation introduced when the page or character was photographed at an angle; (5) resizing to a fixed 64×64 resolution; and (6) normalization of pixel values to the [0, 1] range to stabilize gradient-based training. The pipeline exposes both an in-memory array function, for the web application and API to preprocess an uploaded image without writing a temporary file to disk, and a file-path function that reads an image and calls the same array function — so the two entry points are guaranteed to behave identically. This directly follows the precedent set for Nepali OCR by Hengaju and Bal, whose eight-step preprocessing pipeline for Tesseract-OCR demonstrated that image conditioning by itself — independent of the recognition model — recovers a substantial share of achievable accuracy on real Nepali document photos [14].

For the Transformer branch specifically, one further preprocessing step was required and is discussed in Section 3.5.3: TrOCR's pretrained image encoder expects dark ink on a light background, whereas the shared pipeline above produces a white-on-black binary image (inverse thresholding is standard for CNN training but is the opposite convention TrOCR was pretrained on).

3.3 Data Augmentation

Data augmentation was applied to the training split only, never to validation or test data, so that reported evaluation metrics reflect performance on genuinely unseen, undistorted samples. For the character-level Transformer model, mild affine augmentation (rotation up to 5°, scaling 0.9-1.1×, translation up to 5% of image size, shear up to 5°) was applied; the transformations were kept deliberately small because a Devanagari character's identity depends on the exact shape and position of its shirorekha and matras, so aggressive distortion risks turning one valid character into a plausible rendering of a different one. For the document-level synthetic data (Section 3.6.1), a wider range of handwriting-like augmentation — shear/slant, rotation, Gaussian blur, speckle noise, and contrast jitter — was applied during image generation itself, since the goal there was to simulate the range of real handwriting and scan conditions a line-recognition model would encounter, not merely to regularize a classifier against overfitting a small set of clean glyphs.

3.4 OCR Model Architecture

3.4.1 CNN-Based Model: CRNN with CTC

The CNN-based baseline follows the CRNN design established by Shi, Bai, and Yao [7]: a convolutional feature extractor produces a sequence of feature columns from the input image, a Bidirectional LSTM models dependencies across that sequence in both directions, and a final linear layer produces a per-time-step probability distribution over the 46 character classes plus the CTC blank symbol, trained with Connectionist Temporal Classification loss so that no explicit per-character segmentation is required during training. The convolutional stack extracts a hierarchy of visual features directly from the 64×64 preprocessed input — edges and strokes in early layers, increasingly character-specific shapes in later layers — the same mechanism that gives modern CNN classifiers their accuracy on isolated Devanagari glyphs (Chapter 2) [2], [3]. Decoding at inference time uses greedy CTC decoding: at each time step the most probable class is taken, consecutive repeated predictions are collapsed to one, and the blank symbol is stripped, yielding the final predicted character sequence. The trained network has 6,712,559 parameters.

3.4.2 Transformer-Based Model: TrOCR

The Transformer-based model fine-tunes `microsoft/trocr-base-handwritten`, following the TrOCR architecture of a pre-trained Vision Transformer image encoder paired with a pre-trained Transformer text decoder [10]. The encoder splits the input image into fixed-size patches, embeds each patch, adds positional encodings, and processes the resulting sequence with self-attention exactly as described in Chapter 2 [9]. The decoder generates output text autoregressively: at each step, self-attention over the tokens generated so far and cross-attention over the full sequence of encoded image patches jointly determine the next character, which is what allows the same architecture to be extended from single characters (this section) to full lines (Section 3.6) purely by increasing the maximum generation length, without any change to the model's structure. Because the recognition targets are real Devanagari glyphs rather than opaque numeric class indices, target text for the Transformer model was mapped from each DHCD class folder name to its actual Unicode character (for example, the class `character_1_ka` was mapped to क) using a dedicated label-mapping table, so the Transformer's output is directly human-readable Nepali text rather than a class ID that must be looked up separately.

3.5 Model Training

3.5.1 CRNN Training

The CRNN model was trained with the Adam optimizer and a step-decay learning-rate schedule, with gradient clipping to prevent occasional large CTC-loss gradients from destabilizing training, and early stopping on validation loss to avoid overfitting once the model stopped improving. Batch size was set to 64 when training on a GPU and reduced to 16 for CPU-only runs. Training was carried out on a Kaggle Tesla T4 GPU; the run early-stopped after 22 epochs, with the best validation loss (0.0448) reached at epoch 12, and the corresponding checkpoint — rather than the final epoch's weights — was kept for evaluation. Per-epoch training and validation loss were logged to a CSV file to allow the learning curve to be inspected after training (Chapter 4).

One implementation bug was found and fixed during evaluation of this model: the greedy CTC decoder's original implementation used a heuristic that compared the length of the time-step axis against the size of the character set to decide how the model's output tensor was laid out, and this heuristic silently produced the wrong layout for any batch size greater than one, collapsing predictions across the whole batch and driving measured accuracy toward zero. The fix replaced the heuristic with an explicit `time_first` argument, defaulting to the CRNN's actual native output layout of (time, batch, classes), removing the ambiguity entirely rather than trying to detect it. This bug is reported here because it is a useful, concrete illustration of a broader lesson in sequence-model evaluation: an evaluation script's own decoding logic can silently produce a near-0% accuracy result that looks like a model-training failure but is in fact a bug in the evaluation code, not the trained model — a distinction that only becomes visible by manually inspecting a handful of individual predictions rather than trusting an aggregate metric alone.

3.5.2 TrOCR Training (Single Character)

The Transformer model was fine-tuned with the AdamW optimizer, again with best-checkpoint selection on validation loss and early stopping, using the same train/validation/test split as the CRNN model (Section 3.1) so the two models never saw a different sample of the data. Configurable environment overrides were provided for model choice, batch size, number of epochs, a maximum-training-samples cap, and learning rate, so that a smaller run could be executed quickly on CPU to validate the data pipeline before committing to a full GPU training run — this smaller, network-free validation pass (checking the label map, preprocessing, and augmentation logic without downloading the pretrained model or running on real hardware) passed before any GPU time was spent.

3.5.3 Debugging TrOCR: Two Root-Caused Training Bugs

The first full GPU training run of the character-level TrOCR model, on a Kaggle T4 over three epochs on the full dataset, produced a result that looked like a training failure at first: 0% accuracy, a Character Error Rate and Word Error Rate of 1.0 (i.e., every character and every word wrong), and a validation loss around 9.8 — close to the loss expected from a model outputting tokens at random over a roughly 50,000-token vocabulary — even though the teacher-forced training loss had fallen to a reasonable 0.40. A large gap between a low teacher-forced training loss and a catastrophic free-running generation result is itself diagnostic: it points at the generation (inference-time decoding) path rather than at the model's learned weights, since teacher forcing feeds the model the correct previous token at every step during training loss computation, masking any problem that only manifests when the model must condition on its own (possibly wrong) previous output.

Root-causing this pointed to two independent bugs, both specific to how a pretrained Transformer's own conventions were configured, rather than to the CRNN-vs-Transformer comparison itself:

(1) Wrong decoder start token. The training script had overridden the decoder's start-of-sequence token to the `cls` token (index 0), but `microsoft/trocr-base-handwritten`'s own pretrained configuration uses the `sep` token (index 2) as its decoder start token. Generating from a start token the pretrained decoder had never actually been conditioned on during its own pretraining pushed the very first generation step out of the distribution the decoder was trained on, and the model responded by immediately emitting an end-of-sequence token — producing an empty prediction for every input, which is exactly the systematic 0% accuracy / CER 1.0 / WER 1.0 pattern observed. This was fixed by setting `sep_token_id` as the decoder start token on both the model configuration and the generation configuration, matching the value documented in the base model's own `generation_config.json`.

(2) Inverted image polarity. The shared preprocessing pipeline (Section 3.2) produces an inverse-binary image — white foreground strokes on a black background — because that is the standard convention for training a CNN from scratch. `microsoft/trocr-base-handwritten`, however, was pretrained on natural photographs of handwriting, which are dark ink on light paper. Feeding the inverted, white-on-black convention into a pretrained Vision Transformer encoder corrupted the visual features it computed, since every convolutional and attention filter in the pretrained encoder had learned to respond to dark-on-light patterns specifically. This is consistent with why the validation loss was already near-random from the very first training epoch, before the decoder-start-token bug alone would fully explain: the encoder itself was operating far outside the input distribution it was pretrained on. The fix inverted the image polarity specifically for Transformer inputs — dark ink on light background — while leaving the CRNN's inputs on their original inverse-binary convention, since the CRNN is trained entirely from scratch and has no pretrained expectation to match.

Both fixes were verified independently: the corrected decoder-start-token value was cross-checked against the base model's own published configuration rather than assumed, and the polarity fix was verified by visually inspecting a sample of the images actually being fed to the Transformer processor. A full retrain with both fixes in place is the pending step that produces this project's final single-character Transformer accuracy, CER, and WER figures (see the placeholders and outstanding-work list in Chapter 4).

3.6 Extending to Document-Level Recognition

3.6.1 Motivation and the Synthetic-Data Strategy

Testing the character-level models' natural extension to a "document mode" — segmenting a page into characters and classifying each one — on a real photograph of continuous handwriting exposed the structural limitation described in Chapter 1: because the shirorekha visually joins the letters of a word into a single connected stroke, character-segmentation can only cut cleanly at whitespace between words, and each resulting word-sized blob was then handed to a model that can only output one base glyph. A sample line reading "मेरो नाम संस्कृति हो।" was recognized under this approach as the nonsense string "त्रगल३४ ज". This confirmed that improving the single-character classifier further would not resolve the problem: what was needed was a sequence model trained directly on whole word and line images, following the same segmentation-free principle already established for the character-level CRNN (Section 3.4.1) and consistent with the CRNN-CTC precedent already demonstrated specifically for Nepal script by Nakarmi et al. [15].

Because no sufficiently large public corpus of labelled, real handwritten Nepali lines exists (Chapter 2) [18], this project adopted the standard "synthetic pretraining" recipe used elsewhere in low-resource OCR: render synthetic word and line images from digital Devanagari fonts with handwriting-like distortion, and train the sequence model on this synthetic corpus rather than on scarce real data.

3.6.2 Synthetic Data Generator

A synthetic Nepali line-image generator renders text using Devanagari fonts (Nirmala UI on Windows, with automatic detection of Mangal or Noto Sans Devanagari where present, and automatic detection of Linux-available Devanagari fonts such as Lohit Devanagari when run on a cloud GPU environment) and applies handwriting-like augmentation — shear/slant, rotation, blur, speckle noise, contrast jitter, and randomized margins — so that a single piece of digital text produces many visually distinct training images. The generator's vocabulary mixes real Nepali words with randomly generated valid syllables, and produces matras, halant-joined conjuncts, independent vowels, and digits and punctuation directly, addressing exactly the character categories the isolated single-character dataset (Section 3.1) does not contain. Each generated image is paired with its ground-truth text string in a labels file.

The generator was first built to produce short phrases only (one to four tokens per line). After the first trained checkpoint (Section 3.6.5) was tested against a real printed Nepali government-style form and found to collapse specifically on long lines — six or more tokens, mixing Devanagari and Latin content (for example "Original ID/URL:-"), numbered or bulleted list markers, and form-field labels ending in ":-" — the generator was extended with a dedicated long-line mode: lines of five to thirteen tokens, an optional leading list marker (१./•/✓) on roughly two-fifths of long lines, occasional Latin tokens interspersed among Devanagari words, slash-joined tokens, and a trailing form-label or sentence-terminating punctuation mark. The generator's built-in Nepali vocabulary was simultaneously extended with official-document and government-form terminology (श्रीमान्, निवेदन, प्रहरी, सम्बन्धमा, कार्यालय, दस्तखत, and related terms) that the original, more conversational word list did not cover, since the target use case is government and institutional document digitization (Chapter 1). A command-line flag controls what fraction of a generated dataset uses the long-line mode versus the original short-phrase mode, so both regimes are represented in a single training set.

3.6.3 Line Segmentation

A dedicated line-segmentation function was implemented that returns one bounding box per text line — the union of the bounding boxes of every character detected on that line — rather than one box per character. This sidesteps the shirorekha problem entirely at the segmentation stage: because the function never attempts to separate the individual letters of a word, the joined stroke that breaks character-level segmentation is simply never encountered, and the sequence model downstream is given a whole line to read rather than a pre-broken fragment of one.

3.6.4 Word/Line-Level Dataset and Training

A dedicated dataset class reads (image, full-text) pairs from the labels file produced by the synthetic generator, applies a deterministic seeded train/validation/test split so that no separate split file needs to be maintained for synthetic data, and feeds each image to the TrOCR processor without the 64×64 single-character preprocessing used for isolated glyphs — line and word images are variable in size and are already rendered dark-ink-on-light-paper, the same domain the pretrained TrOCR encoder expects (Section 3.5.3), so no additional polarity correction is needed for this track. The maximum target sequence length for the tokenizer — the ceiling on how many subword tokens a single training label can be, beyond which the label would be silently truncated — was initially set to 64, sized for short phrases, and was later raised to 160 once the long-line generator (Section 3.6.2) began producing labels with substantially more tokens per line; the inference script imports this same constant, so generation and training are always kept in sync automatically rather than needing to be updated in two places. Training itself reuses the same model-configuration and epoch-loop logic already validated for the character-level TrOCR model (Section 3.5.2), with the generation length raised to match, and writes its checkpoint to a separate directory so a word-level training run can never overwrite the single-character checkpoint.

3.6.5 Training Runs and a GPU-Quota Lesson

An initial word-level TrOCR checkpoint was trained on a Kaggle T4 GPU using the original short-phrase-only synthetic data. The resulting weights were integrated into the document-digitization web application (Section 3.7) and confirmed to produce real, readable Devanagari output on held-out synthetic samples, though — as described in Section 3.6.2 — testing against an actual printed government-style form subsequently revealed the long-line failure mode that motivated the generator improvements.

A second training run, using 12,000 images generated with the improved long-line-aware generator over eight epochs, reached a best validation loss of 1.8006 at the final epoch. This run illustrated a practical constraint of Transformer-based OCR flagged in Chapter 2 as a general limitation of the architecture family [9]: partway through the session, the cloud notebook's GPU usage quota was exhausted before the trained checkpoint had been exported out of the ephemeral training session, and the session's working files — including the trained weights — were not recoverable once the session was recycled. The first checkpoint (previous paragraph) therefore remained the version integrated into the running system pending a repeat of this training run. The concrete process change adopted as a result — export and download the checkpoint immediately once training finishes, before running any further cells in the same session — is recorded here because it is a genuine methodological lesson from this project's own execution, not a generic best practice restated from a textbook.

3.7 Document Digitization Web Application

The trained models were exposed through a self-contained web application (Flask backend, plain HTML/CSS/JavaScript frontend, no separate build step) that implements the actual document-digitization goal stated in Chapter 1: converting a scanned page into editable, searchable Unicode text. A document-processing endpoint accepts a single image or a multi-page PDF and returns per-page recognized text together with the bounding boxes of each recognized line; it selects the word-level TrOCR model when a trained checkpoint is present (Section 3.6.5) and otherwise falls back to the character-segmentation CRNN pipeline (Section 3.4.1), so the application degrades gracefully rather than failing outright when the stronger model's weights are unavailable. Recognized text is Unicode-normalized (NFC form) before being returned, so that visually identical Devanagari sequences that could otherwise be encoded as different underlying byte sequences are made consistent.

A separate export endpoint converts the (possibly user-edited) recognized text into plain UTF-8 text, a DOCX file — using the Devanagari-appropriate complex-script font slot so Nepali text renders correctly in Microsoft Word rather than falling back to a default Latin font — or a searchable PDF that overlays the original scanned image with an invisible, selectable Unicode text layer positioned using the recognized line boxes, so the exported PDF looks identical to the original scan but its text can be selected, copied, and searched. One export bug was found and fixed during testing: the text-layer rendering call silently returned a negative value on text overflow rather than raising an exception, so the original overflow-detection logic never triggered and some lines were written with no visible text at all; the fix reduced the computed font size, explicitly checked the call's return value, and fell back to a simpler baseline text-insertion call that cannot silently clip, after which every line in a test document was confirmed to be extractable via copy-paste and via a PDF viewer's find function.

Because the target documents are frequently produced by writers who may find it easier to type in the Preeti legacy font encoding or in romanized Nepali than in native Unicode input methods, two client-side text-conversion tools were built directly into the application: a Preeti-to-Unicode and Unicode-to-Preeti converter implementing the standard Preeti character mapping and its associated visual reordering rules (for example, moving a short-i matra that is typed before its consonant back to its correct logical position after conversion), and a romanized-typing aid that converts Latin-alphabet phonetic input (for example "namaste") into Devanagari Unicode ("नमस्ते") as the user types, so that OCR output can be corrected by a typist without native Devanagari keyboard familiarity.

3.8 Backend API Integration (CNN and Transformer, Behind One Interface)

In parallel with the Flask web application, a FastAPI backend service exposes the same two trained models — CRNN and TrOCR — behind a REST API with persistent storage of every recognition request. An upload endpoint accepts an image or PDF, validates its file type and size, runs the selected model, stores the recognized text together with its confidence score and processing time in a SQLite database, and returns the result; separate endpoints list which models are currently loaded and retrieve past recognition history by ID. Both models are loaded once at application startup rather than per-request, and inference for the Transformer model is delegated to the same word-level prediction module used by the Flask application (Section 3.7) rather than re-implemented a second time, so there is exactly one implementation of TrOCR inference shared by both interfaces. Two integration issues were identified and corrected while wiring this API to the real models rather than the placeholder text it initially returned: the backend's own image-preprocessing code had been duplicated from the shared pipeline (Section 3.2) instead of calling it directly, which is precisely the train/serve-skew risk that pipeline was built to prevent, and a package-naming collision between the backend's own internal module and the top-level model-training package (both named `models`) silently prevented the Transformer model from being imported at all; the backend's internal package was renamed to resolve this. With both fixes applied, the API was confirmed, live, to report both models as loaded and to return real recognized Nepali text — not placeholder text — for both the CRNN and the Transformer model.

3.9 Comparative Analysis

A dedicated comparison script consumes the evaluation output (Section 3.10) of the CRNN and character-level TrOCR models and produces a side-by-side metrics table together with bar-chart comparisons of accuracy/CER/WER and of parameter count/inference latency. Because both models were trained and evaluated on the identical dataset split (Section 3.1) using the identical preprocessing pipeline (Section 3.2) up to the single Transformer-specific polarity step (Section 3.5.3), any difference the comparison reports reflects the two architectures' modelling capability rather than a difference in what data each model saw. The script is written to produce a CNN-only report when Transformer evaluation results are not yet available and to fill in the Transformer columns automatically once they are, which is the current state of this comparison pending the completed Transformer retrain described in Section 3.5.3 (see Chapter 4).

3.10 Model Evaluation

Both models are evaluated using three standard OCR metrics. Accuracy is the proportion of predictions that exactly match the ground truth, computed as the number of correct predictions divided by the total number of predictions. Character Error Rate (CER) measures character-level edit distance between a prediction and its ground truth, normalized by the length of the ground truth:

CER = (S + D + I) / N

where S, D, and I are the number of character substitutions, deletions, and insertions needed to transform the predicted text into the ground-truth text, and N is the number of characters in the ground truth. Word Error Rate (WER) is computed analogously at the word level rather than the character level. CER and WER are reported in addition to accuracy specifically because accuracy alone treats a prediction that is wrong by one substituted character identically to a prediction that is completely unrelated to the ground truth, whereas CER and WER give partial credit for a prediction that is close to correct — a distinction that matters for judging a document-recognition system, where a mostly-correct line is far more useful to a human editor than a completely wrong one, even though both would count as "incorrect" under plain accuracy. Qualitative evaluation was also carried out by direct visual inspection of predicted text against ground truth, specifically to characterize which categories of Devanagari structure (matras, conjuncts, or visually similar consonant pairs) a given error belonged to, rather than relying on the aggregate numeric metrics alone.

3.11 Experimental Environment

Character-level model training (both CRNN and TrOCR) and word-level TrOCR training were carried out on Kaggle's cloud notebook environment using a Tesla T4 GPU, chosen for its availability under Kaggle's free GPU quota and its adequate memory for both a from-scratch CRNN and a fine-tuned base-sized Transformer. Model evaluation, the Flask web application, and the FastAPI backend were run and verified locally on CPU, since CPU inference latency for both trained models (Chapter 4) was low enough for interactive use, and keeping the deployed/demonstration path CPU-only avoids requiring end users or examiners to have GPU hardware available.

3.12 Tools and Technology Stack

The system was implemented in Python throughout. PyTorch was used for both the CRNN (built and trained from scratch) and the TrOCR model (fine-tuned via the Hugging Face Transformers library, which provides the pretrained `microsoft/trocr-base-handwritten` weights, processor, and generation utilities). OpenCV supplied the image-processing primitives (grayscale conversion, blurring, adaptive thresholding, and the affine transform used for deskewing) behind the shared preprocessing pipeline (Section 3.2). The Flask web application and its plain HTML/CSS/JavaScript frontend implement the primary, self-contained document-digitizer demonstration; a parallel FastAPI backend with SQLAlchemy (SQLite) persistence and a React/Vite frontend implement a second, database-backed interface to the same underlying models (Section 3.8). PyMuPDF provides PDF page rendering and the searchable-PDF text-layer overlay, and python-docx provides DOCX export with correct Devanagari complex-script font handling. Pytest is used for the backend's automated test suite (Section 3.14).

3.13 System Design

3.13.1 Context Diagram

At the system boundary, a single external actor — the user — supplies a scanned document image or PDF to the OCR system and receives machine-readable Unicode Devanagari text in return, matching the context diagram used in the project proposal: User → [scanned document] → OCR System → [Unicode Devanagari text] → User.

3.13.2 Data Flow Diagram (Level 1)

Internally, an uploaded document flows through four stages: preprocessing (grayscale, denoising, thresholding, deskew — Section 3.2), line segmentation (Section 3.6.3), text recognition (the character-segmentation CRNN path or the word-level TrOCR path, selected automatically based on checkpoint availability — Section 3.7), and Unicode post-processing (NFC normalization). Three persistent stores support this flow: the labelled character dataset and synthetic line-image dataset used for training (Section 3.1, 3.6.2), the trained model checkpoints for both architectures, and the recognized-text output, which is additionally persisted to a database when accessed through the FastAPI backend (Section 3.8).

3.13.3 Use Case Diagram

The primary actor, a user, can: upload a scanned document or PDF; view the recognized Devanagari Unicode text; edit that text; export it as TXT, DOCX, or a searchable PDF; and use the Preeti-Unicode converter and romanized-typing tools independently of the OCR pipeline itself, since those two tools operate entirely client-side on user-supplied or OCR-produced text. A secondary interaction path, available through the FastAPI backend, allows a user to select which model (CRNN or Transformer) is used for a given request and to retrieve their past recognition history.

3.13.4 Entity Relationship Diagram

The FastAPI backend persists two entities. A Document Image record stores the original filename and the saved file path for each uploaded document. A Recognized Text record stores the model used, the recognized text, its confidence score, its processing time, and a creation timestamp, and references the Document Image it was produced from. This is a deliberately smaller schema than the five-entity design (Document Image, Text Region, Recognized Text, OCR Model, Evaluation Result) sketched in the project proposal: per-line Text Region storage and a separate Evaluation Result table were not implemented, since the Flask web application already returns per-line bounding boxes directly in its API response without needing them persisted relationally, and evaluation results (Chapter 4) are generated and stored as static JSON/CSV artefacts by the offline comparison script (Section 3.9) rather than queried through the live application database.

3.13.5 Sequence Diagram

For a document-recognition request: the user uploads a document through the web interface; the interface forwards the file to the backend; the backend runs preprocessing, then line segmentation, then text recognition (CRNN or TrOCR depending on checkpoint availability); the recognized text is NFC-normalized and, for the FastAPI backend, persisted to the database; and the final text is returned to the interface for display, editing, and optional export.

3.14 Testing and Deployment

Automated testing covers the FastAPI backend with a pytest suite exercising the health check, the model-listing endpoint, a successful OCR request end-to-end (upload through to a stored, retrievable result), rejection of an unsupported file extension, and history retrieval for both an existing and a non-existent record. One test-harness bug was found and fixed during this work: the test client was not being used as a context manager, which meant FastAPI's startup logic — including database table creation — never executed during the test run, causing every test that touched the database to fail with a "no such table" error that had nothing to do with the code under test; wrapping the test client's lifecycle correctly resolved this, after which the full suite passed. The CRNN and character-level TrOCR pipelines each additionally have a fast, dependency-light smoke test that validates the data path (label mapping, preprocessing, and — for TrOCR — augmentation) without requiring network access or a GPU, so that a broken data pipeline is caught before any GPU time is spent training on it. The Flask web application was verified manually end-to-end: random-sample character recognition against known ground truth, full document-mode recognition on a real scanned page, and all three export formats (TXT, DOCX, searchable PDF), with the searchable PDF's text layer confirmed extractable by both copy-paste and a PDF viewer's search function.

Deployment, at the current stage of the project, consists of running the Flask application and the FastAPI backend locally; a Dockerfile is provided for the FastAPI backend to support containerized deployment, and trained model weights are distributed out of version control (via the Kaggle notebook outputs they were trained in) rather than committed to the source repository, since the largest checkpoint exceeds a gigabyte in size. Full cloud deployment of either interface is identified as future work (Chapter 5).
